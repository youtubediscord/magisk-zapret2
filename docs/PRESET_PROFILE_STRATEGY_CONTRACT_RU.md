# Единый контракт «Пресет → Профиль → Стратегия»

Статус: проект целевой архитектуры
Версия контракта: 1
Область действия: root-модуль Zapret2 для Android, `nfqws2`, Android-приложение и обновления модуля

## 1. Назначение

Этот документ задаёт единую модель конфигурации Zapret2 и заменяет параллельные
понятия «категории», «глобальная стратегия», «TXT-пресет» и «custom cmdline».

Целевая модель:

```text
Пресет
├── общие параметры запуска
├── профиль 1
│   ├── имя
│   ├── состояние: включён/выключен
│   ├── фильтры трафика
│   └── стратегия обработки
├── профиль 2
└── ...
```

В нормативных разделах слова **ДОЛЖЕН**, **НЕ ДОЛЖЕН**, **СЛЕДУЕТ** и **МОЖЕТ**
описывают обязательное или рекомендуемое поведение реализации.

## 2. Термины

### 2.1. Пресет

Пресет — полный переносимый набор параметров запуска Zapret2.

Пресет содержит:

- метаданные файла;
- общие параметры `nfqws2`, применяемые ко всему запуску;
- загрузку Lua-файлов и blob-данных;
- упорядоченный список профилей.

В каждый момент времени активен ровно один пресет. Именно он является
единственным источником фильтров и стратегий для нового процесса `nfqws2`.

### 2.2. Профиль

Профиль — одно правило внутри пресета: какой трафик выбрать и как его обработать.

Профиль содержит четыре логические части:

1. имя — `--name=...`;
2. состояние — наличие или отсутствие `--skip`;
3. фильтры и границы обработки;
4. одну стратегию, представленную упорядоченной цепочкой `--lua-desync`.

Профили выполняются в порядке, записанном в пресете, и разделяются отдельной
строкой `--new`.

### 2.3. Стратегия

Стратегия — конкретный способ обработки уже выбранного профилем трафика.

В формате `nfqws2` стратегия — одна или несколько строк `--lua-desync=...` внутри
одного профиля. Несколько таких строк образуют одну упорядоченную цепочку, а не
несколько независимых профилей.

Стратегия `--lua-desync=pass` является допустимой явной стратегией «ничего не
изменять».

Готовая стратегия из каталога — только шаблон для редактора. При выборе её
упорядоченные строки `--lua-desync` копируются в профиль активного пресета.
Идентификатор каталога не записывается в пресет и никогда не передаётся
`nfqws2`. Поэтому каталог не является вторым источником runtime-конфигурации.

### 2.4. Общие параметры

Общие параметры располагаются до первого `--name=...`. Они не принадлежат
первому профилю и применяются ко всему процессу.

К ним относятся, например:

- `--lua-init=...`;
- `--blob=...`;
- `--ctrack-disable=...`;
- другие явно разрешённые глобальные параметры Android-сборки `nfqws2`.

Параметры жизненного цикла и изоляции модуля — номер NFQUEUE, mark, UID/GID,
pidfile — образуют системную оболочку запуска. Они принадлежат модулю, а не
пользовательской стратегии, и безопасно добавляются вокруг активного пресета.

Все пользовательские и файловые blob-объявления, необходимые стратегиям
пресета, находятся в этом глобальном блоке самого TXT. Внешнего общего каталога
blob-объявлений нет. Встроенные имена blob-ов самого `nfqws2` не требуют строки
`--blob` и не считаются отдельным источником конфигурации.

Все 98 встроенных пресетов содержат одинаковый полный канонический blob-блок.
Импортёр строит его как непротиворечивое объединение объявлений исходного набора,
сортирует по имени blob и физически записывает в каждый TXT. Два разных значения
одного имени являются ошибкой импорта. Это правило гарантирует доступность
готовых стратегий без runtime-обращения к `blobs.txt` или иному реестру.

## 3. Единственный источник истины

### 3.1. Активный запуск

Целевая версия `runtime.ini` хранит только ссылку на активный пресет и системные
параметры модуля:

```ini
[core]
active_preset=Default v1 (game filter).txt
autostart=1
wifi_only=0
qnum=200
desync_mark=0x40000000
nfqws_uid=0:0
log_mode=none
```

`active_preset` — единственный ключ выбора источника профилей. Переходные алиасы
и одновременная поддержка нескольких ключей не допускаются.

Грамматика и типы runtime-v1 принадлежат `scripts/runtime-config.sh` и
используемому им lifecycle-парсеру. Kotlin редактирует текст, но не дублирует
семантическую валидацию: каждый кандидат проверяется модулем до атомарной
публикации. Обычное чтение никогда не восстанавливает и не перезаписывает файл.
Отсутствующее, небезопасное, повреждённое и несовместимое состояние возвращается
как разные типизированные результаты. Явный repair выполняется под общей
блокировкой мутаций, создаёт канонический `[core]` из встроенных значений и
сохраняет не-core секции; Android и boot lifecycle используют один shell-путь
восстановления.

Все модульные ошибки передаются в APK одним ограниченным envelope:

```text
schema=1
status=ERROR
domain=CONFIG
stage=RUNTIME_PARSE
code=INVALID_QNUM
detail=qnum=70000, expected 1..65535
```

`domain`, `stage` и `code` являются непрозрачными строками: APK проверяет только
форму, версию и лимиты, поэтому новые значения отображаются без пересборки.
`detail` однострочный и ограничен 512 байтами. Envelope является UI-сводкой и не
заменяет полный lifecycle/logcat/файловый журнал.

### 3.2. Пресет по умолчанию

Обязательный встроенный пресет первого запуска:

```text
Default v1 (game filter).txt
```

Используется существующий файл из каталога `zapret2/presets`, приведённый
импортёром к Android-контракту версии 1. Его имена профилей, порядок, фильтры,
селекторы, `--skip` и стратегии сохраняются; Windows-only параметры и
`--ipcache*` удаляются согласно разделам 7–8.

Требования:

- fresh install записывает это точное имя в `active_preset`;
- файл входит в обязательную часть package contract;
- release build проверяет структуру, зависимости и Android-совместимость файла;
- device preflight выполняет его `nfqws2 --dry-run` до первого применения;
- ошибка проверки блокирует сборку или запуск и не выбирает другой пресет;
- встроенный файл является обычным рабочим пресетом: профильный и полный
  текстовый редакторы изменяют именно выбранный TXT на месте, без автоматического
  создания копии и без смены `active_preset`;
- установка следующей версии модуля перезаписывает этот файл новой комплектной
  версией без merge и без сохранения локальных изменений.

### 3.3. Удаляемые параллельные источники

Runtime контракта версии 1 НЕ ДОЛЖЕН читать:

- `preset_mode`;
- `strategy_preset`;
- `categories.ini`;
- `strategies-tcp.ini`;
- `strategies-udp.ini`;
- `strategies-stun.ini`;
- `blobs.txt`;
- отдельный режим `custom_cmdline_file`.

Для этих источников не создаются fallback, конвертеры или ветки совместимости.
Файлы `strategies-tcp.ini`, `strategies-udp.ini`, `strategies-stun.ini` и
`blobs.txt`, а также их одноразовые конвертеры удаляются из release-пакета в той
же версии, в которой исчезают последние production-ссылки на них. Оставлять их
как скрытый fallback или параллельный каталог запрещено.

### 3.4. Запрет неявного fallback

Если активный пресет отсутствует, небезопасен, несовместим или не проходит
`nfqws2 --dry-run`, запуск завершается ошибкой. Runtime НЕ ДОЛЖЕН молча переходить
на категории, другой пресет или встроенную глобальную стратегию.

## 4. Канонический формат файла

### 4.1. Кодировка и границы

- UTF-8 без BOM;
- переводы строк LF;
- не более 1 МиБ;
- не более 256 профилей;
- обычный файл без symlink и hardlink;
- имя файла — безопасное непосредственное имя внутри каталога пресетов.

Рекомендуемый заголовок:

```text
# Preset: Default v1
# Preset-Format: 1
# Source: builtin
```

Комментарии и пустые строки разрешены и сохраняются редактором.

### 4.2. Общая грамматика

```text
<metadata and comments>
<global option>*

--name=<human-readable label>
[--skip]
<profile filter>+
<profile selector>*
<lua strategy>+

--new

--name=<human-readable label>
[--skip]
<profile filter>+
<profile selector>*
<lua strategy>+
```

Ограничения:

- первый профиль начинается с `--name=...` без предшествующего `--new`;
- между соседними профилями находится ровно один `--new`;
- `--new` не может быть первой или последней значимой строкой;
- пустые профили запрещены;
- все профильные параметры находятся после имени соответствующего профиля;
- после начала первого профиля глобальные параметры запрещены.

Канонический формат использует отдельные `--name=...` и `--new`. Сокращение
`--new=name`, хотя и поддерживается upstream, в сохраняемых файлах версии 1 не
используется: у профиля должен быть один очевидный источник имени.

## 5. Контракт профиля

### 5.1. Имя: `--name`

Каждый профиль ДОЛЖЕН содержать ровно одну строку:

```text
--name=<имя профиля>
```

Требования к имени:

- после удаления внешних пробелов имя не пустое;
- не более 128 символов и 512 байт UTF-8;
- отсутствуют NUL, CR, LF и управляющие символы;
- пробелы, кириллица и обычная пунктуация разрешены.

Имя является отображаемой меткой, а не идентификатором профиля. Одинаковые
`--name` внутри одного пресета разрешены и сохраняются без автоматического
переименования: такие имена уже встречаются в каноническом исходном наборе.
Редактор адресует профиль его позицией в разобранном snapshot и ожидаемым
SHA-256 всего файла, а не поиском по имени.

`--name` является рабочей опцией `nfqws2`, а не комментарием только для UI.
Сборщик ДОЛЖЕН передать всё значение как один аргумент argv. Запрещено удалять
имя, делить его по пробелам или заменять без необходимости подчёркиваниями.

### 5.2. Состояние: `--skip`

Включённый профиль не содержит `--skip`.

Выключенный профиль содержит ровно одну отдельную строку:

```text
--skip
```

Правила:

- `--skip` относится только к текущему профилю;
- выключение не удаляет фильтры или стратегию;
- повторное включение удаляет только `--skip`;
- предпочтительное каноническое место — сразу после `--name`;
- `--skip` передаётся `nfqws2`, который штатно игнорирует такой профиль;
- выключенный профиль не участвует в расчёте Android-перехвата трафика.

### 5.3. Фильтры

Фильтры отвечают только на вопрос «какой трафик относится к профилю».

К ним относятся:

- `--filter-l3=...`;
- `--filter-tcp=...`;
- `--filter-udp=...`;
- `--filter-l7=...`;
- `--hostlist=...`;
- `--hostlist-domains=...`;
- `--hostlist-exclude=...`;
- `--ipset=...`;
- `--ipset-exclude=...`.

Профиль ДОЛЖЕН иметь хотя бы один явный положительный фильтр. Намеренный профиль
для всего TCP или UDP задаётся явно (`--filter-tcp=*` или `--filter-udp=*`), а не
получается из отсутствия фильтров.

Android-правила NFQUEUE ДОЛЖНЫ охватывать консервативное объединение трафика всех
включённых профилей. Фильтр в `nfqws2` не может расширить более узкий kernel
capture. Реализация НЕ ДОЛЖНА поддерживать отдельный ручной список портов,
способный незаметно разойтись с активным пресетом.

Если протокол невозможно однозначно получить из фильтра профиля, адаптер Android
ДОЛЖЕН потребовать явную область перехвата или признать пресет несовместимым.

#### 5.3.1. Политика kernel capture

TXT является источником не только port union и argv `nfqws2`, но и внешних
ограничителей `connbytes`, требуемых upstream Linux-интеграцией. До первого
глобального аргумента каждый пресет ДОЛЖЕН ровно один раз объявить:

```text
# NFQWS2_TCP_PKT_OUT=20
# NFQWS2_TCP_PKT_IN=10
# NFQWS2_UDP_PKT_OUT=20
# NFQWS2_UDP_PKT_IN=10
```

Это Android-метаданные, отражающие четыре одноимённых upstream-настройки. Каждое
значение — каноническое положительное десятичное число не длиннее девяти цифр.
Компилятор включает их в аутентифицированный argv artifact, owner metadata,
firewall spec, fingerprint и WAL. `runtime.ini` и Control не имеют второй копии
этих значений.

Политика задаётся явно, потому что `--out-range`/`--in-range` допускают счётчики
`n`, `d`, `b`, `s`, `p`, а также `a` и `x`; общего корректного преобразования
такого диапазона в kernel packet count не существует. Значения выбираются
согласно требованиям всех включённых профилей пресета. Недостаточный bound
считается дефектом пресета, а избыточный увеличивает CPU-нагрузку.

### 5.4. Селекторы и границы обработки

Следующие параметры уточняют, когда применять стратегию к уже выбранному трафику:

- `--out-range=...`;
- `--payload=...`;
- другие поддержанные `nfqws2` селекторы профиля.

Они принадлежат профилю, а не стратегии и не глобальной конфигурации.

### 5.5. Стратегия

Профиль ДОЛЖЕН содержать хотя бы одну строку:

```text
--lua-desync=<обработчик и параметры>
```

Все `--lua-desync` текущего профиля образуют одну стратегию и сохраняют порядок.
Внешняя ссылка на имя стратегии из `strategies-*.ini` запрещена: исполняемая
стратегия полностью записана в самом профиле.

### 5.6. Каталоги готовых стратегий

Вместо старых `strategies-*.ini` целевая поставка содержит четыре неизменяемых
каталога шаблонов редактора:

```text
zapret2/strategy-catalogs/
├── tcp.txt
├── udp.txt
├── voice.txt
└── http80.txt
```

Исходной базой служат четыре переданных одноимённых файла. Области каталогов:

- `tcp.txt` — TCP/TLS;
- `udp.txt` — UDP/QUIC;
- `voice.txt` — STUN, Discord и другой голосовой UDP-трафик;
- `http80.txt` — незашифрованный HTTP по TCP/80.

UI автоматически выбирает каталог только по разобранным фильтрам профиля:

1. `voice.txt` — профиль содержит `--filter-l7` с токеном `stun` или `discord`,
   либо это UDP-профиль с Discord hostlist/ipset;
2. `http80.txt` — профиль является только TCP и его разрешённое множество портов
   равно `80`;
3. `udp.txt` — остальные однозначно UDP-профили;
4. `tcp.txt` — остальные однозначно TCP-профили.

Имя из `--name` не участвует в классификации. Смешанный или неоднозначный
профиль не получает каталог по догадке и возвращает машинную причину
`STRATEGY_CATALOG_SCOPE_UNRESOLVED`. Компилятор не меняет профиль по имени
каталога. Одинаковые ID в разных каталогах допустимы: полная идентичность
шаблона — пара `<catalog, id>`. Внутри одного файла ID обязан быть уникальным.

Формат одной записи:

```text
[stable_id]
name = Человекочитаемое имя
author = Автор
label = recommended
description = Описание
blobs = tls_google,quic1
--lua-desync=handler:arg=value
--lua-desync=next_handler:arg=value
```

Правила формата:

- `[id]`, `name` и хотя бы одна непустая строка `--lua-desync` обязательны;
- `author`, `label`, `description` и `blobs` — метаданные интерфейса;
- `label` не влияет на запуск;
- несколько `--lua-desync` образуют одну упорядоченную цепочку;
- `args=...`, фильтры, селекторы, `--new`, `--name`, `--skip`, shell-синтаксис и
  `--ipcache*` в каталоге запрещены;
- `blobs` является подсказкой интерфейсу, а не доверенным списком: валидатор
  вычисляет фактические blob-зависимости из самой цепочки;
- неизвестный metadata-ключ, дублирующийся ключ или дублирующийся ID делает
  каталог невалидным; строка стратегии с неизвестным обработчиком или
  неразрешимым blob делает невалидной конкретную запись.

Каталоги проверяются при сборке и при чтении приложением. В picker попадают
только записи, для которых доступны все Lua-обработчики и blob-зависимости в
глобальном блоке активного пресета. Разрешённые встроенные blob-имена `nfqws2`
проверяются отдельно и не требуют объявления. Отсутствующая файловая зависимость
не берётся из `blobs.txt` и не добавляется молча: шаблон показывается недоступным
с точной причиной или не показывается в обычном режиме. Выбрать такой шаблон
нельзя.

Переданная база нормализуется до включения в release:

- из `http80.txt` удаляется второй, побайтно одинаковый блок
  `[multisplit_midsld]`;
- отдельные проверяемые `fakemultisplit.lua` и `fakemultidisorder.lua` входят в
  Android-пакет и явно загружаются глобальным блоком каждого встроенного пресета,
  поэтому восемь соответствующих TCP-записей проверяются наравне с остальными;
- записи со ссылками на preset-local blob-имена, которых нет в активном пресете,
  остаются недоступными;
- никакой дубликат, отсутствующая зависимость или неизвестный обработчик не
  исправляется подбором похожей стратегии.

Выбор готовой стратегии заменяет только строки `--lua-desync` целевого профиля.
Имя, `--skip`, match-фильтры, `--payload`, `--in-range`, `--out-range`, комментарии
и общие параметры пресета не меняются. Если зависимости новой цепочки не
разрешены до записи, операция завершается ошибкой без изменения TXT.

После raw-редактирования UI сравнивает фактическую упорядоченную цепочку
`--lua-desync` с каталогом. Точное совпадение может показываться именем готовой
стратегии; любое другое содержимое показывается как `Custom`. Отдельно хранить
выбранный ID запрещено.

## 7. Android-совместимость

### 7.1. Сохраняемые опции

`--name` и `--skip` являются переносимыми рабочими опциями и НЕ ДОЛЖНЫ удаляться
импортёром, валидатором или сборщиком команды.

Windows-only параметры (`--wf-*`, WinDivert-зависимости и неподдержанные Lua-
обработчики) не передаются Android-бинарнику. Их удаляет или преобразует только
явный импортёр до сохранения Android-пресета.

Live-валидатор не должен молча пропускать неизвестную значимую опцию: такой файл
помещается в карантин с конкретной причиной.

### 7.2. Политика IP cache

Все опции с префиксом `--ipcache` запрещены на Android:

```text
--ipcache-hostname
--ipcache-hostname=...
--ipcache-lifetime=...
--ipcache...
```

Требования:

- импортёр удаляет их из импортируемых Windows-пресетов;
- валидатор сохранённого или пользовательского пресета отклоняет их;
- custom/import migration отклоняет или удаляет их до активации;
- финальная граница сборки повторно проверяет, что `--ipcache` не попал в argv;
- запрет не реализуется молчаливым игнорированием уже активного файла.

Следствие этого решения: нулевая фаза TCP (`syndata`) не получает доменное
ограничение от одного `--hostlist`, потому что имя хоста ещё неизвестно на SYN.
Пресет и его порядок обработчиков сохраняются дословно, но UI и документация не
должны обещать host-scoped `syndata` без отдельного `--ipset`. Автоматическая
адаптация не удаляет такие обработчики и не добавляет скрытый `ipcache`.

### 7.3. Зависимости

Допустимы только объявленные обычные файлы внутри разрешённых каталогов модуля:

- `@lua/<file>`;
- `@bin/<file>`;
- `lists/<file>`.

Перед запуском путь разрешается в абсолютный путь установленного модуля.
Запрещены `..`, абсолютные пользовательские пути, symlink, hardlink, пустые,
нечитаемые, слишком большие и не объявленные package manifest зависимости.

## 8. Импорт Windows-пресетов

Импорт является отдельной детерминированной операцией, а не частью live-запуска.

Импортёр ДОЛЖЕН:

1. прочитать исходный пресет без изменения порядка профилей;
2. сохранить `--name`, `--skip`, фильтры, селекторы и `--lua-desync`;
3. удалить `--ipcache*`;
4. удалить Windows-only глобальные параметры;
5. отклонить профиль, который невозможно безопасно адаптировать для Android;
6. проверить все зависимости;
7. привести структуру к каноническому формату версии 1;
8. добавить явную Android capture policy в TXT;
9. прогнать тот же валидатор, который используется перед запуском;
10. сформировать отчёт о каждом удалённом или отклонённом элементе.

Импортёр НЕ ДОЛЖЕН превращать именованные профили в безымянные блоки и НЕ ДОЛЖЕН
скрывать несовместимость простым удалением неизвестных строк.

### 8.1. Зафиксированная исходная база

Перед реализацией runtime-контракта встроенный каталог готовится из точных
источников:

- имена и структура winws2-пресетов — ревизия
  `youtubediscord/zapret@eb98b501acd22623d1570be28574c0110f10df11`;
- Android-списки — переданный архив `lists.zip` с SHA-256
  `e76409252f438d8847891c62b504f706884803f5aebb2235b93c89eedbf810da`;
- upstream-контракт `nfqws2`, использованный при подготовке этой исходной базы, —
  release `v1.0.3`, commit
  `b78b52c4cd7f843da3ff0848a3430afbd401bdf2`. Это историческая база импорта,
  а не pin сборки: каждая CI-сборка получает latest stable upstream Release,
  записывает выбранные tag/commit/digest и повторно квалифицирует полный каталог.

Из архива списков импортируются только прямые `.txt`-зависимости пресетов.
Записи `Zone.Identifier`, каталоги и иные побочные файлы не входят в пакет.
Кодировка приводится к UTF-8 без BOM, переводы строк — к LF; каждая hostlist- и
ipset-строка проходит проверку своего типа. Историческое имя
`russia-youtube-rtmps.txt`, использованное как ipset, канонизируется в
`ipset-russia-youtube-rtmps.txt`, потому что содержимое файлов совпадает.

Отсутствующий blob не подменяется другим пакетом. Подготовка release-каталога
останавливается до получения и проверки всех требуемых файлов: заведомо
quarantined-пресеты не считаются допустимым окончательным результатом реализации.
Переданный `bin.zip` добавляет `@bin/quic_initial_dbankcloud_ru.bin` и закрывает
последнюю файловую зависимость исходного набора. Все 98 встроенных пресетов,
включая обязательный `Default v1 (game filter).txt`, классифицируются compatible;
перед релизом они ДОЛЖНЫ дополнительно пройти общий валидатор и реальный
`nfqws2 --dry-run`.

## 9. Валидация и компиляция

### 9.1. Этапы валидации

Один валидатор используется при сканировании каталога, импорте, сохранении,
применении и запуске:

1. проверка безопасного файла и размера;
2. лексический разбор строк;
3. структурная проверка global/profile/`--new`;
4. проверка ровно одного непустого `--name` и не более одного `--skip` в профиле;
5. проверка фильтров и стратегии каждого профиля;
6. проверка Android-совместимости опций;
7. разрешение и проверка зависимостей;
8. построение kernel capture для включённых профилей;
9. построение точного argv;
10. `nfqws2 --dry-run`.

Неизвестная, потерянная или неоднозначно распарсенная значимая строка приводит к
ошибке. Частично собранный пресет не запускается.

### 9.2. Построение argv

Сборщик создаёт типизированную последовательность аргументов, а не одну строку,
которая затем повторно делится shell по пробелам.

Upstream-загрузка параметров через единственный аргумент `@config_file` не
поддерживается Android-сборкой `nfqws2`. Поэтому Android runtime НЕ ДОЛЖЕН
запускать `nfqws2 @preset` и не должен менять или форкать бинарник только ради
этого механизма.

Вместо этого используются два разных файла с разной ответственностью:

- исходный TXT-пресет — единственный редактируемый источник истины;
- приватный служебный argv-артефакт — результат компиляции для одного запуска.

Служебный артефакт создаётся заново при каждом apply, принадлежит root, доступен
только владельцу и находится в управляемом runtime-каталоге. Пользователь и UI
его не редактируют. Он содержит точные границы нормализованных аргументов,
разрешённые абсолютные пути и module-owned системные параметры.

Launcher читает артефакт без `eval`, command substitution и повторного shell-
парсинга. Каждая запись добавляется как один argv-элемент. После построения argv
launcher повторно проверяет запрет `--ipcache`, размер argv и целостность
артефакта.

Логическая форма запуска:

```text
nfqws2
  <module-owned system arguments>
  <preset global arguments>
  <profile 1 arguments>
  --new
  <profile 2 arguments>
  ...
```

Границы каждого аргумента должны сохраняться дословно. В частности,
`--name=Голосовые звонки/чаты` является одним argv-элементом.

В диагностический файл публикуется воспроизводимое escaped-представление argv,
а ownership metadata сверяет фактический `/proc/<pid>/cmdline`.

Один и тот же скомпилированный набор профильных аргументов используется сначала
для dry-run, затем для реального запуска. Различаться могут только явно
определённые служебные аргументы `--dry-run`, `--daemon` и `--pidfile`.

### 9.3. Свежесть исходника и артефакта

Компиляция привязана к точному snapshot исходного пресета. Snapshot включает как
минимум безопасный путь, metadata файла и SHA-256 содержимого.

Порядок операции:

1. прочитать и захэшировать активный TXT-пресет;
2. разобрать и проверить его;
3. построить firewall plan и argv-артефакт;
4. повторно проверить metadata и SHA-256 исходника;
5. атомарно опубликовать артефакт;
6. выполнить dry-run;
7. ещё раз подтвердить связь `active_preset → source hash → artifact hash`;
8. применить firewall и запустить процесс;
9. сверить фактический argv и состояние firewall.

Если исходник или указатель активного пресета изменился на любом этапе, операция
завершается `PRESET_SOURCE_CHANGED`. Устаревший артефакт никогда не запускается.

## 10. Операции приложения

### 10.1. Экран пресетов

Экран пресетов отвечает за цельные наборы запуска:

- просмотр каталога;
- выбор активного пресета;
- импорт;
- полный текстовый редактор для эксперта;
- просмотр точного итогового `nfqws2` argv для текущего текста редактора.

Редактор имеет две связанные вкладки: исходный TXT и итоговая команда. Preview
строится из текущего черновика, в том числе ещё не сохранённого, тем же безопасным
компилятором и проходит `nfqws2 --dry-run`. Он показывает executable, границы
каждого аргумента, точные TCP/UDP-множества перехвата и служебные `--daemon` /
`--pidfile`, когда их поддерживает установленный бинарник. Preview не меняет
`active_preset`, постоянный TXT или состояние сервиса; временный root-owned файл
удаляется до публикации результата в UI.

### 10.2. Экран профилей

Экран профилей показывает профили активного пресета в порядке исполнения:

- имя;
- состояние;
- краткое описание фильтра;
- краткое описание стратегии;
- включение/выключение через `--skip`;
- редактирование;
- изменение порядка перемещением цельного блока профиля.

Отдельный экран выбора «стратегии для категории» в новой модели не нужен.

### 10.3. Изменения

Любая операция над активным пресетом выполняется под общей mutation lock:

1. получить авторитетный snapshot файла и активного указателя;
2. построить новое содержимое в памяти;
3. проверить новым единым валидатором;
4. атомарно записать файл;
5. выполнить dry-run;
6. если сервис работал — перезапустить его; если он был остановлен — сохранить
   остановленное состояние;
7. для работавшего сервиса проверить новый процесс и firewall;
8. при ошибке восстановить snapshot и предыдущий рабочий запуск.

Успех публикуется только после post-operation verification.

### 10.4. Синхронизация TXT и экрана профилей

Экран профилей не имеет собственного постоянного хранилища фильтров, стратегий
или состояния. Его модель всегда является проекцией текущего TXT-файла,
указанного в `active_preset`.

Правила чтения:

- при открытии экрана и после смены активного пресета файл читается заново;
- после любого события успешного сохранения/apply экран заново читает файл;
- возврат из полного текстового редактора всегда инвалидирует профильный cache;
- cache разрешён только как оптимизация с ключом `path + SHA-256`, но не как
  источник данных;
- изменение имени, фильтра, стратегии, `--skip`, порядка или числа профилей в
  полном редакторе должно отобразиться без перезапуска приложения.

Правила записи из экрана профилей:

1. экран запоминает SHA-256 прочитанного исходника;
2. изменение применяется к типизированной модели профиля;
3. сериализуется весь пресет с сохранением не затронутых строк и комментариев;
4. repository сравнивает ожидаемый SHA-256 с текущим файлом;
5. при несовпадении возвращает `PRESET_SOURCE_CHANGED` и ничего не перезаписывает;
6. при совпадении выполняет полную валидацию и атомарную запись;
7. persisted-файл читается повторно, и UI публикует только эту модель;
8. если пресет активен, запускается транзакционный apply.

Операции имеют точную область изменения:

- выключение добавляет только `--skip` после `--name`;
- включение удаляет только `--skip` текущего профиля;
- переименование меняет только `--name`;
- смена hostlist/ipset меняет только соответствующие match-сегменты;
- готовая стратегия заменяет только упорядоченные строки `--lua-desync`, но не
  имя, состояние, фильтры, селекторы или общие параметры;
- перестановка переносит профиль целиком вместе с комментариями и сохраняет
  корректные разделители `--new`.

После сохранения raw-редактор и экран профилей обязаны показывать одно и то же
содержимое. Отдельная таблица выбранных стратегий, флагов enabled или фильтров
запрещена.

## 11. Граница совместимости

Контракт версии 1 вводится как новая реализация только для Zapret2. Существование
старой модели, старой установки или Zapret1 не входит в область реализации;
миграционный код не создаётся.

Требования:

- legacy `runtime.ini`, `categories.ini`, `strategies-*.ini`, старый custom
  cmdline, старые TXT, hostlist и Lua не ищутся, не читаются, не копируются, не
  архивируются и не преобразуются;
- при первом запуске версии с новым контрактом создаётся новый `runtime.ini` и
  выбирается валидный встроенный пресет по умолчанию;
- в production-коде отсутствуют legacy-парсеры, конвертеры, dual-read и fallback.

Это намеренно несовместимое изменение конфигурации. Простота и однозначность
нового runtime имеют приоритет над сохранением старых настроек.

## 12. Обновления и пользовательские данные

- встроенные пресеты поставляются модулем и имеют известную версию;
- встроенный пресет редактируется на месте в пределах установленной версии;
- следующее обновление безусловно заменяет все встроенные TXT новыми комплектными
  версиями; локальные изменения встроенных файлов не объединяются и не
  восстанавливаются;
- явно импортированные пользовательские пресеты с отдельными именами и активный
  указатель сохраняются при совместимом обновлении, если указатель остаётся
  валидным;
- обновление сначала валидирует активный пресет против нового `nfqws2` и его Lua;
- несовместимость блокирует применение или сохраняет предыдущий рабочий релиз,
  но не включает другой пресет молча.

## 14. Критерии готовности реализации

Контракт считается реализованным, когда одновременно выполняются условия:

1. fresh install выбирает `Default v1 (game filter).txt` единственным указателем
   активного пресета;
2. `categories.ini`, `strategies-*.ini`, `blobs.txt`, `strategy_preset` и cmdline
   mode не влияют на argv и не читаются production-кодом;
3. каждый профиль имеет ровно один непустой `--name`, при этом одинаковые
   отображаемые имена допустимы;
4. toggle профиля изменяет только наличие `--skip` и перезапускает сервис
   транзакционно;
5. `--name` с пробелами и кириллицей доходит до `nfqws2` одним argv-элементом;
6. `--skip` доходит до `nfqws2` и выключенный профиль не расширяет NFQUEUE capture;
7. `--ipcache*` удаляется импортёром и отклоняется всеми live-входами;
8. фильтры и `--lua-desync` берутся только из активного пресета;
9. firewall является проверенным superset включённых профилей;
10. неизвестные опции не пропускаются молча;
11. любой save/apply проходит один валидатор и `nfqws2 --dry-run`;
12. ошибка записи, dry-run, restart или verification восстанавливает предыдущий
    рабочий пресет и состояние сервиса;
13. GUI показывает точный итоговый argv для сохранённого текста и несохранённого
    черновика без изменения активной конфигурации.

## 15. Обязательная матрица проверок

Реализация не считается готовой без автоматических host-тестов и проверок на
реальном Android-устройстве.

Минимальные сценарии:

1. первый запуск выбирает `Default v1 (game filter).txt`, а последующий выбор
   пресета согласованно меняет `active_preset`, экран профилей и итоговый argv;
2. raw-редактор меняет `--name`, и новое имя появляется на экране профилей;
3. raw-редактор добавляет, удаляет и переставляет профили, UI повторяет структуру;
4. raw-редактор добавляет/удаляет `--skip`, toggle показывает правильное состояние;
5. toggle в UI меняет только `--skip`, а raw-редактор видит тот же результат;
6. выбор готовой стратегии меняет только строки `--lua-desync` профиля, а после
   raw-редактирования точное совпадение снова определяется из TXT без скрытого ID;
7. изменение hostlist/ipset меняет match и пересчитывает firewall plan;
8. имя с пробелами и кириллицей остаётся одним элементом в `/proc/<pid>/cmdline`;
9. изменение файла между read и write возвращает `PRESET_SOURCE_CHANGED`;
10. изменение файла между compile и launch блокирует устаревший артефакт;
11. неизвестная опция, unsafe dependency и `--ipcache*` блокируют apply;
12. выключенный профиль остаётся в argv с `--skip`, но не расширяет firewall;
13. dry-run и реальный процесс используют один скомпилированный профильный argv;
14. ошибка restart/verification восстанавливает предыдущий файл, argv и firewall;
15. многократный parse → serialize без изменений даёт побайтно тот же TXT;
16. закрытие и повторное открытие приложения восстанавливает UI только из
    `active_preset` и его TXT, без скрытого состояния.
17. четыре новых каталога проходят проверку уникальности ID, грамматики,
    обработчиков и blob-зависимостей; дубликат и недоступная зависимость дают
    детерминированную ошибку.
18. `strategies-tcp.ini`, `strategies-udp.ini`, `strategies-stun.ini`, `blobs.txt`
    и их production-ссылки отсутствуют в release-пакете.
19. два профиля с одинаковым `--name` редактируются по позиции независимо и не
    переименовываются автоматически.
20. удаление всех `--blob` из тестового пресета делает зависящую от них стратегию
    невалидной и не активирует fallback на общий каталог.
21. все 98 встроенных пресетов содержат один и тот же полный канонический
    blob-блок, а конфликт двух определений одного имени блокирует импорт.
22. picker автоматически открывает `voice` для STUN/Discord, `http80` для чистого
    TCP/80, `udp` для остального UDP и `tcp` для остального TCP.
23. обновление заменяет локально изменённый встроенный TXT новой комплектной
    версией без merge.
24. preview несохранённого TXT проходит единый компилятор и dry-run, сохраняет
    аргумент с пробелами одним элементом, показывает точный TCP/UDP union и всегда
    удаляет временный candidate.

Device-проверка дополнительно подтверждает PID ownership, фактический argv,
iptables/ip6tables, работу `--name`, `--skip` и сетевой трафик хотя бы для одного
TCP-, UDP- и L7-профиля.

## 16. Нормативные внешние основания

- Upstream `nfqws2`: точный stable release выбирается на каждой сборке и
  записывается в package provenance:
  <https://github.com/bol-van/zapret2/releases/latest>
- Upstream `nfqws2`: `--new` начинает новый профиль, `--name` задаёт имя, а
  `--skip` игнорирует профиль; CI читает manual выбранного release:
  <https://github.com/bol-van/zapret2/blob/v1.0.3/docs/manual.md>
- Upstream `nfqws2`: `@config_file` явно не поддерживается Android-сборкой;
  Android использует собственный compiler + launcher:
  <https://github.com/bol-van/zapret2/blob/v1.0.3/docs/manual.md#общие-принципы-задания-параметров>
- Формат именованных профилей, используемый исходными winws2-пресетами:
  <https://github.com/youtubediscord/zapret/tree/eb98b501acd22623d1570be28574c0110f10df11/src/presets/builtin/winws2>
